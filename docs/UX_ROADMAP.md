# UX Roadmap — "사용자가 쉽게 쓸 수 있는 S3Desk" 로드맵

> 이 문서는 S3Desk를 **처음 접하는 사용자가 5분 이내에 핵심 흐름을 완료**할 수 있도록 하기 위해 필요한 작업을 분석하고 우선순위를 매긴 로드맵입니다.
>
> 기존 `docs/ROADMAP.md`(기술 로드맵)와 보완 관계이며, 이 문서는 **사용자 관점의 경험 개선**에 집중합니다.

---

## 현재 상태 요약

### 잘 되어 있는 것 ✅

| 영역 | 현재 상태 |
|------|-----------|
| 접근성(a11y) | Skip link, `aria-label`, keyboard navigation, ARIA live region 구현 완료 |
| 에러 UX | `normalizedError` 기반 공통 에러 코드 + 권장 조치 텍스트 제공 |
| 네트워크 상태 | 오프라인/불안정 배너 자동 표시 |
| SPA 내비게이션 | React Router 기반, 풀 리로드 없는 라우팅 완료 |
| 번들 최적화 | Light shell(/profiles)에서 antd 미로딩, 지연 로딩으로 초기 로드 ~85 kB gzip |
| 다운로드/업로드 fallback | Presigned → proxy/staging 자동 fallback |
| Provider 가이드 | Profile 모달에서 provider별 필수 필드 검증 + 문서 링크 |

### 개선이 필요한 것 ❌

| 영역 | 현재 상태 | 사용자 영향 |
|------|-----------|------------|
| 온보딩 | 첫 방문 시 체크리스트만 표시, 해제하면 사라짐 | 초보자가 다음 단계를 모름 |
| 용어 설명 | "Profile", "Transfer engine", "presigned" 등 설명 없음 | 기술 배경 없으면 혼란 |
| 빈 상태 안내 | "No buckets yet" 수준의 메시지 | 왜 비어있는지, 뭘 해야 하는지 모름 |
| 인라인 도움말 | 폼 필드에 placeholder만 존재, 툴팁/설명 없음 | 어떤 값을 넣어야 할지 모름 |
| 설정 접근성 | API Token 설정이 Settings drawer 안에 숨어있음 | 첫 설정 시 찾기 어려움 |
| 모바일 최적화 | 레이아웃은 반응형이나 터치 UX 미검증 | 모바일에서 CTA 접근 불편 |
| 피드백 루프 | Job 완료 시 toast만 표시 | 성공/실패를 놓치기 쉬움 |

## Phase 1 — 첫 사용 5분 이내 완료 (First-Time User Experience)

> **목표**: 처음 접속한 사용자가 도움 없이 "Profile 생성 → Bucket 조회 → Object 다운로드" 흐름을 완료.

### 1-1) 환영/소개 화면 (Welcome Screen)

- **위치**: `/profiles` (프로필이 0개이고, 한 번도 해제하지 않은 경우 표시)
- **내용**:
  - S3Desk가 무엇인지 한 문장 소개 ("클라우드 저장소를 웹에서 관리하는 대시보드")
  - 3단계 핵심 흐름 시각화: ① 프로필 만들기 → ② 버킷 탐색 → ③ 파일 업로드/다운로드
  - "시작하기" CTA 버튼 → Profile 생성 모달로 이동
- **기대 효과**: 사용자가 앱의 목적과 다음 행동을 즉시 이해
- **작업량**: S (프론트엔드 컴포넌트 1개)

### 1-2) 대화형 프로필 생성 가이드 (Guided Profile Creation)

- **현재**: Profile 모달이 provider 선택 후 즉시 모든 필드를 표시
- **개선**: 스텝 단위 진행 (Step 1: Provider 선택 → Step 2: 필수 정보 → Step 3: 연결 테스트)
  - Step 1에서 provider별 설명과 아이콘 제공 (AWS S3, Azure, GCS 등)
  - Step 2에서 필수 필드만 먼저 표시, 고급 옵션은 접기
  - Step 3에서 자동 연결 테스트 실행 + 성공/실패 안내
- **기대 효과**: 초보 사용자도 provider 지식 없이 첫 프로필 생성 가능
- **작업량**: M (기존 ProfileModal 리팩토링)

### 1-3) 필드별 인라인 도움말 (Inline Help)

- **현재**: `endpoint`, `region`, `accessKeyId` 등에 placeholder만 존재
- **개선**: 각 필드 아래 또는 옆에 짧은 설명 텍스트 + "더 보기" 링크
  - 예: `Endpoint` → "스토리지 서버의 주소입니다. MinIO: `http://host:9000`, AWS: 비워두면 기본값 사용"
  - 예: `Region` → "버킷이 위치한 리전입니다. 예: `us-east-1`, `ap-northeast-2`"
  - 예: `API Token` → "이것은 S3 키가 아닙니다. 서버 시작 시 설정한 API_TOKEN 값입니다."
- **기대 효과**: 외부 문서 참조 없이 폼 완성 가능
- **작업량**: S (FormField extra prop 활용)

### 1-4) 연결 실패 시 문제 해결 가이드 (Connection Troubleshooting)

- **현재**: "Connection test failed" + 에러 코드만 표시
- **개선**: 에러 코드별 구체적 안내 표시
  - `invalid_credentials` → "Access Key 또는 Secret Key가 올바른지 확인하세요"
  - `endpoint_unreachable` → "Endpoint URL이 올바른지, 서버에서 접근 가능한지 확인하세요"
  - `signature_mismatch` → "Region이 실제 버킷 위치와 일치하는지 확인하세요"
- **기대 효과**: 연결 실패 시 스스로 문제 해결 가능
- **작업량**: S (에러 코드 → 가이드 텍스트 매핑)

## Phase 2 — 핵심 워크플로우 간소화 (Core Workflow Simplification)

> **목표**: 반복적으로 수행하는 작업(업로드, 다운로드, 검색, Job 관리)을 최소 클릭으로 수행.

### 2-1) 빈 상태(Empty State) 개선

- **현재**: "No buckets yet", "No files selected" 등 단순 텍스트
- **개선**: 빈 상태에서 다음 행동을 안내하는 컨텍스트 메시지 + CTA
  - Buckets 빈 상태 → "아직 버킷이 없습니다. 버킷을 생성하거나, 스토리지에 이미 버킷이 있다면 연결 설정을 확인하세요." + [New Bucket] [Profiles 확인]
  - Objects 빈 상태 → "이 버킷은 비어 있습니다." + [Upload Files] [Create Sync Job]
  - Jobs 빈 상태 → "실행 중인 작업이 없습니다. 파일을 업로드하거나 동기화 작업을 만들어 보세요." + [Uploads] [New Job]
- **기대 효과**: 다음 행동이 명확해져 이탈 감소
- **작업량**: S (각 Page에 EmptyState 컴포넌트 추가)

### 2-2) 퀵 액션 바 (Quick Actions)

- **현재**: 각 기능이 페이지별로 분산 (업로드는 Uploads 탭, Job은 Jobs 탭)
- **개선**: Objects 페이지 상단에 자주 쓰는 액션 바
  - [Upload] [Download Selected] [Delete Selected] [New Job] [Search]
  - Keyboard shortcut 힌트 표시 (예: `Ctrl+U` Upload)
- **기대 효과**: 주요 액션 접근 시간 감소
- **작업량**: M (Objects 페이지 toolbar 확장)

### 2-3) 드래그 & 드롭 업로드 개선

- **현재**: Uploads 페이지에서만 파일 선택 가능
- **개선**: Objects 페이지에서 직접 드래그 & 드롭 업로드
  - 드래그 시 오버레이 표시: "여기에 파일을 놓으면 현재 폴더에 업로드합니다"
  - 드롭 후 Transfers drawer 자동 열림
- **기대 효과**: 별도 Uploads 탭 이동 없이 즉시 업로드
- **작업량**: M (Objects 페이지 DnD 핸들러 + Transfers 연동)

### 2-4) Job 상태 알림 개선

- **현재**: Job 완료 시 toast만 표시
- **개선**:
  - 브라우저 Notification API 연동 (탭 비활성 시 데스크톱 알림)
  - JobQueueBanner에 최근 완료/실패 요약 표시
  - 실패 Job에 "재시도" 버튼 직접 표시
- **기대 효과**: 백그라운드 작업 결과를 놓치지 않음
- **작업량**: S (Notification API + 배너 확장)

## Phase 3 — 발견 가능성 및 학습 곡선 완화 (Discoverability & Learning)

> **목표**: 사용자가 매뉴얼 없이 고급 기능을 발견하고 활용.

### 3-1) 컨텍스트 툴팁 시스템

- **현재**: 일부 버튼에만 Tooltip 존재
- **개선**: 모든 주요 기능 옆에 `(?)` 아이콘 + 간략한 도움말 팝오버
  - "Dry Run" → "실제 변경 없이 어떤 파일이 영향받는지 미리 확인합니다"
  - "Force Path Style" → "S3 호환 스토리지(Ceph, MinIO)에서 주로 필요합니다"
  - "Presigned Upload" → "브라우저에서 스토리지로 직접 업로드합니다. CORS 설정이 필요할 수 있습니다"
- **작업량**: S (공통 HelpTooltip 컴포넌트)

### 3-2) 설정 탐색 개선

- **현재**: Settings는 drawer 안에 flat하게 나열
- **개선**:
  - 설정 항목을 카테고리별 섹션으로 그룹화 (연결, 전송, 보안, 고급)
  - 각 설정 항목에 간단한 설명 텍스트
  - 검색/필터 기능 (설정이 많아질 때 대비)
- **작업량**: M (SettingsDrawer 구조 리팩토링)

### 3-3) 키보드 단축키 가이드

- **현재**: 일부 단축키 존재하나 발견 불가
- **개선**:
  - `?` 또는 `Ctrl+/` 로 단축키 모달 열기
  - 단축키 목록: Navigate (G+B Buckets, G+O Objects, G+J Jobs, G+U Uploads)
  - 주요 액션 (Ctrl+U Upload, Delete 선택 삭제, Enter 열기)
- **작업량**: S (KeyboardShortcutGuide 모달)

### 3-4) 인앱 릴리스 노트 / 기능 하이라이트

- **현재**: 새 기능 추가 시 알림 없음
- **개선**:
  - 앱 버전 업데이트 감지 시 "새 기능" 배지 표시
  - 클릭하면 최근 변경사항 요약 모달
- **작업량**: S (버전 비교 + 모달)

## Phase 4 — 고급 사용 편의성 (Power User Experience)

> **목표**: 반복 작업을 자동화하고, 대규모 운영에서도 효율적으로 사용.

### 4-1) Job 템플릿 / 즐겨찾기

- **현재**: 매번 Job 파라미터를 처음부터 입력
- **개선**:
  - 이전 Job 설정을 "다시 실행" 가능
  - 자주 쓰는 Job 설정을 템플릿으로 저장
  - 템플릿에서 "실행" 클릭 시 파라미터 미리 채움
- **작업량**: L (Backend: 템플릿 CRUD API + Frontend: 템플릿 UI)

### 4-2) 대시보드 / 한눈에 보기

- **현재**: 각 기능이 탭별로 분리되어 전체 현황 파악 어려움
- **개선**: Dashboard 페이지 추가
  - 활성 프로필 요약 (provider, 버킷 수)
  - 진행 중 / 최근 Job 요약 (실행 중 N개, 실패 N개)
  - 스토리지 사용량 개요 (provider가 지원하는 경우)
  - 최근 활동 타임라인
- **작업량**: L (새 페이지 + 집계 API)

### 4-3) 대량 작업 UX 개선

- **현재**: 50,000개 키 삭제 시 전체 키를 수동 선택
- **개선**:
  - "전체 선택" + prefix 기반 필터 적용 후 대량 액션
  - 작업 예상 시간 표시
  - 진행 중 배치 작업의 단계별 상태 시각화 (준비 → 실행 → 정리)
- **작업량**: M (Objects 선택 로직 + Job 시각화)

### 4-4) 다국어 지원 (i18n)

- **현재**: 영어 기반 UI (일부 한국어 문서)
- **개선**:
  - i18n 프레임워크 도입 (react-intl 또는 i18next)
  - 최소 범위: 한국어, 영어
  - 사용자 설정에서 언어 선택
- **작업량**: L (전체 UI 문자열 추출 + 번역 파일)

## 검증 기준 (Definition of Done)

각 항목별 완료 기준:

- **Phase 1** (첫 사용 경험)
  - 신규 사용자가 문서 없이 프로필 생성 → 버킷 조회까지 5분 이내 완료 (수동 테스트)
  - Playwright E2E: 온보딩 흐름 시나리오 추가
  - `npm run lint && npm run test:unit && npm run build` 통과

- **Phase 2** (핵심 워크플로우)
  - 주요 태스크(업로드/다운로드/검색) 완료에 필요한 클릭 수 30% 감소 (비교 측정)
  - 빈 상태에서 다음 액션까지 도달 시간 10초 이내
  - Playwright smoke 전 시나리오 통과

- **Phase 3** (발견 가능성)
  - 고급 기능(Dry Run, Policy Editor 등) 사용률 증가 (향후 analytics 연동 시)
  - 모든 주요 설정 항목에 인라인 설명 존재
  - 키보드 단축키로 주요 내비게이션 100% 가능

- **Phase 4** (고급 사용)
  - 반복 Job 생성 시간 50% 감소 (템플릿 활용 시)
  - Dashboard에서 전체 현황을 한 화면에 확인 가능

## 기존 로드맵과의 관계

| 기존 로드맵 (ROADMAP.md) | 이 로드맵 (UX_ROADMAP.md) |
|--------------------------|--------------------------|
| P0-P3 기술 품질 (CI, 테스트, 번들, a11y) | Phase 1-4 사용자 경험 |
| 코드/인프라 관점 | 사용자/행동 관점 |
| "에러를 정규화한다" | "에러가 발생하면 다음에 뭘 하라고 알려준다" |
| "번들을 줄인다" | "로딩이 빨라서 이탈하지 않는다" |
| "a11y 컴플라이언스" | "실제로 키보드/스크린리더로 쓸 수 있다" |

**원칙**: 기술 로드맵이 UX 로드맵의 기반입니다. 기술적 기반(에러 정규화, a11y, 번들 최적화)이 완료된 현재, UX 로드맵은 그 기반 위에 사용자 가시적 개선을 쌓는 단계입니다.
