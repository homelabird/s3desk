- 구조는 이전 배치들 덕분에 많이 정리됐지만, 현재 남은 핵심 부채는 "큰 중앙 orchestration 축"과 "정책 문서/CI scope의 마지막 10%"에 집중돼 있습니다.
- 예전 상위 이슈였던 `apiToken` query 허용은 현재 [middleware.go](/home/homelab/Downloads/project/s3desk/backend/internal/api/middleware.go)에서 이미 차단되고 있어 우선순위에서 내려도 됩니다.
- 지금 가장 현실적인 위험은 다음 네 축입니다.
  - frontend page/hook orchestration 과집중
  - backend realtime / restore path 신뢰성
  - CI scope hole 및 evidence drift
  - 기본 green gate가 놓치는 provider/OS-specific 검증 공백

## P1 이슈

### 1. Bundle Budget runtime scope hole

- 근거:
  - [.github/workflows/frontend-e2e.yml](/home/homelab/Downloads/project/s3desk/.github/workflows/frontend-e2e.yml)
- 문제:
  - `frontend/package.json`이 `frontend_bundle_contract`에는 있었지만 `frontend_bundle_runtime`에는 없었습니다.
  - 결과적으로 dependency/runtime bundle 변화가 있어도 `npm run bundle:budget`가 skip될 수 있었습니다.
- 영향:
  - 실제 번들 회귀가 `Bundle Budget Contract`만 green이고 runtime budget은 확인되지 않는 상태가 생길 수 있습니다.
- 권고:
  - `frontend/package.json`을 runtime scope에 포함해 dependency-level 변화가 항상 runtime budget lane을 타도록 고정

### 2. `Jobs` page controller가 아직 중앙 orchestration hotspot

- 근거:
  - [useJobsPageController.tsx](/home/homelab/Downloads/project/s3desk/frontend/src/pages/jobs/useJobsPageController.tsx)
- 문제:
  - query, route-state prefill, localStorage, device upload/download bridge, realtime, table layout, overlay state가 한 훅에 과하게 모여 있습니다.
- 영향:
  - 작은 기능 추가도 query/regression/layout 영역을 함께 건드리게 되고, 테스트 계약이 있어도 수정면이 큽니다.
- 권고:
  - `queries`, `create flows`, `layout/presentation state` 축으로 2~3개 controller 훅으로 추가 분리

### 3. realtime event 전달이 slow consumer에서 조용히 유실됨

- 근거:
  - [hub.go](/home/homelab/Downloads/project/s3desk/backend/internal/ws/hub.go)
  - [handlers_events.go](/home/homelab/Downloads/project/s3desk/backend/internal/api/handlers_events.go)
  - [handlers_ws.go](/home/homelab/Downloads/project/s3desk/backend/internal/api/handlers_ws.go)
- 문제:
  - `Hub.Publish`가 client channel overflow 시 `default:`로 drop하고 끝납니다.
  - reconnect/resume에서도 sequence gap을 명시적으로 알려주지 않습니다.
- 영향:
  - UI가 `job.created`, `job.completed`, progress 이벤트를 놓쳐 "작업이 멈춘 것처럼 보이는" stale 상태를 만들 수 있습니다.
- 권고:
  - overflow를 metric/log로 남기고, overflow client를 `reconnect_required` 또는 `sequence_gap` semantics로 처리

### 4. restore/import preflight가 temp spool 이후에 일어남

- 근거:
  - [handlers_server_backup.go](/home/homelab/Downloads/project/s3desk/backend/internal/api/handlers_server_backup.go)
  - [handlers_server_portable.go](/home/homelab/Downloads/project/s3desk/backend/internal/api/handlers_server_portable.go)
- 문제:
  - multipart bundle이 temp 영역에 먼저 스풀된 뒤에야 staging/preflight가 실행됩니다.
- 영향:
  - low-disk 환경에서는 앱이 의도한 preflight보다 먼저 `/tmp`/quota 오류가 터질 수 있습니다.
- 권고:
  - handler에서 `ParseMultipartForm` 의존을 줄이고 server-managed staging path로 직접 스트리밍하도록 경계 재설계
